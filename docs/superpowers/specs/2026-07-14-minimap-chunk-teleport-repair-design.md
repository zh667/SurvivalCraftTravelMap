# Survivalcraft Travel Map：小地图、整区块探索与传送修复设计

日期：2026-07-14

状态：用户已逐段确认，待实现计划

适用分支：`feature/travel-map`

## 1. 背景与实机证据

第二轮隔离世界测试确认：大地图可由 `M` 打开，但右上角小地图体量过大、玩家传送文字按钮破坏 HUD 一致性；探索区域虽然能够补绘，却按不完整边缘逐渐出现；两次集成主机地表传送均返回 `InternalError`，玩家未发生移动。

当前测试设置保存了 `MiniMapSize=384`。在 1440×960、当前游戏 UI 缩放下，这约等于 512–518 个屏幕像素；现有参考小地图的视觉体量约为 256×256 屏幕像素，因此当前地图约大一倍。

探索链路目前在玩家换到任意新方格时，以 `VisibilityRange / 2` 为半径逐列扫描。实测可视距离为 64，每轮扫描 65×65＝4225 列。游戏地形区块为 16×16；World2 已触及的 68 个区块中只有 30 个完整，另有 38 个只记录了一部分，证明逐列方形扫描与用户期望的整区块揭示不一致。

传送日志记录了两个集成主机请求，结果均为 `InternalError`。静态核对真实 `Survivalcraft.dll` 后确认：`SurvivalcraftPlayerFacade` 只用 `BindingFlags.NonPublic` 查找移动状态字段，而实际 `ComponentHealth.m_wasStanding` 是公开字段。第一次调用 `CaptureSnapshot()` 时类型初始化失败，玩家尚未移动，异常随后被主机请求的通用 `catch` 隐藏。

## 2. 已确认的产品决策

- 小地图采用“原版风格的自适应版”：参考简洁体量和游戏原生观感，但继续使用逻辑 UI 坐标适配分辨率与缩放。
- 打开背包、角色面板、制作台、睡眠界面、任意模态对话框或大地图时，小地图和玩家传送入口暂时隐藏；关闭后自动恢复，不改变持久设置。
- 探索采用“亲自进入才揭示”：玩家进入新的 16×16 区块后，在地表数据就绪的第一帧一次显示整个区块；不提前揭示相邻区块。
- 重新进入已探索区块时重新采样完整区块，以刷新地形变化并自动修复旧的残缺或透明缓存。
- 地图坐标、坐标点和玩家邀请继续共用服务器权威安全传送服务。

### 2.1 组件边界

- `TravelMapHudPolicy`：只根据运行模式、主玩家、设置、模态界面和其他玩家数量计算地图与传送图标的可见性，不读取或写入地图数据。
- `MiniMapRenderer`：只负责小地图视觉和输入；通过既有回调请求打开大地图，不直接管理 Dialog 生命周期。
- `TerrainChunkExplorationScheduler`：把玩家位置转换为 16×16 区块键，维护唯一待办并决定何时重试，不直接持久化像素。
- `ExplorationRecorder.RecordChunk`：完成 256 列临时采样和原子提交；底层 `ExplorationTileStore` 继续负责 64×64 瓦片、版本和 flush。
- `SurvivalcraftTerrainMapSource`：封装真实地形状态和列采样，向调度器暴露明确的区块地表就绪能力。
- `SurvivalcraftPlayerFacade`：封装真实玩家移动字段绑定；`SafeTeleportService` 保持与游戏反射细节隔离。
- `TeleportDiagnosticReporter`：接收传送阶段与异常并分别输出日志和用户友好结果，不参与传送决策。

## 3. 小地图 UI

### 3.1 尺寸与位置

- 默认逻辑尺寸从 256 改为 `192×192`；在当前 UI 缩放下约为 256×256 屏幕像素。
- 保留 `160 / 192 / 256 / 320 / 384` 五档尺寸，满足用户主动放大或缩小的需求。
- 设置 schema 升级；由于当前仍是未公开测试构建，旧 schema 中的测试值 `384` 一次迁移为 `192`，其他受支持尺寸保持不变。
- 使用 `GuiWidget.ActualSize` 的逻辑坐标，默认顶部边距 24、右侧边距 76；换算后接近原界面的 32 与 100 屏幕像素留白，并避开右侧游戏按钮列。
- 窗口或 UI 缩放改变后，在下一次布局更新中重新定位。

### 3.2 视觉语言

- 未探索背景使用中性深灰黑，不再使用大面积墨绿色面板。
- 外框使用 2 逻辑像素暖灰白边线，外侧增加 1 像素半透明暗影，使地图在明暗地形上均可辨认。
- 玩家标记为约 18 逻辑像素的红色方向三角，附细暗描边；小地图不再绘制青色十字准星。
- 航点绘制为 5–6 逻辑像素菱形；名称仅在大地图显示，避免小地图文字拥挤。
- 坐标位于地图内部左下角半透明窄条，格式为 `X:488 Y:63 Z:-60`，使用约 0.65 的辅助字体比例。
- 日夜明暗只作用于地形像素，不改变边框、玩家箭头、航点和坐标文字。

### 3.3 交互与 HUD 可见性

- `M` 继续打开/关闭大地图。
- 单击或轻触小地图也可打开大地图；滚轮缩放仍只在指针位于小地图时生效。
- 删除小地图上方的“玩家传送”文字按钮。
- 单人世界或当前没有其他玩家时，不显示玩家传送入口。
- 联机且存在其他玩家时，在小地图下方靠右、间隔 4 逻辑像素显示 `48×46` 的原生斜面图标按钮，屏幕体量约为 64×62 像素。
- HUD 可见性由运行时界面状态统一计算。地图设置关闭、非主玩家、大型/模态界面、大地图或世界卸载中的任一条件成立时，小地图与传送按钮均不可见、不可点击。
- 普通聊天消息显示时地图可保留；文本输入正在占用焦点时只禁用地图交互。

## 4. 进入区块后一次性揭示

### 4.1 区块身份与触发

- 新增独立的 16×16 地形区块坐标值对象，世界坐标使用数学向下取整划分；必须正确覆盖 `-1 / -16 / -17` 等负坐标边界。
- 玩家进入世界时登记当前区块。
- 每帧比较玩家当前区块与上一帧区块；发生变化时只把新进入的区块加入待处理集合。
- 相同区块只保留一个待办，避免边界抖动或连续更新造成重复工作。
- 传送到远处后，位置变化同样触发目标区块登记。

### 4.2 就绪与调度

- 地图探索更新顺序调整到地形更新之后，使本帧刚完成的地表数据可以立即被看到。
- 待处理区块每帧检查，不再受静止 0.5 秒周期限制。
- 当真实区块状态达到 `TerrainChunkState.InvalidPropagatedLight` 或更高时，地表内容、高度及环境颜色输入已经可读，可以开始采样；不再等待 `InvalidVertices1 / InvalidVertices2 / Valid` 这些最终网格阶段。
- 尚未就绪、存储压力或采样异常都不得移除待办，也不得写入部分结果；世界卸载时取消待办。

### 4.3 原子整块提交

- 一个待办先把完整 16×16＝256 列采样到临时缓冲区。
- 只有 256 列全部取得合法非透明颜色后，才在一次提交中写入探索存储。
- 任一列暂不可用时丢弃本轮临时缓冲并在后续帧重试，保证屏幕上不会出现半个区块。
- 提交在内存中立即递增地图版本；下一次绘制显示整个区块。现有定期 flush 仅负责磁盘持久化，不影响实时显示。
- 每次重新进入区块都重新采样并覆盖这 256 列，因此旧的部分探索位、透明像素和之后发生的地形变化都会自动刷新。
- 现有 64×64 `.sctm` 持久化瓦片格式保持不变；一个持久化瓦片容纳 4×4 个游戏地形区块，不要求删除或迁移世界地图文件。
- 每次跨区块只采样 256 列，明显少于旧实现单轮 4225 列。

## 5. 传送根因修复与诊断能力

### 5.1 字段绑定修复

- 将移动状态字段查询改为 `BindingFlags.Instance | BindingFlags.Public | BindingFlags.NonPublic`。
- 提供单一的 `GetRequiredBooleanInstanceField` 验证入口：字段必须存在、必须为实例字段且类型必须为 `bool`。
- 继续兼容字段在不同联机版中公开或私有的变化，不直接假设单一可见性。
- 由于失败的静态类型初始化会污染当前进程，安装修复包后必须完全退出并重启游戏再复测。

### 5.2 安全事务保持不变

- 地表传送由服务器使用目标 X/Z 重新加载区块、计算安全 Y 并搜索安全落点。
- 坐标点传送只在保存 Y 的上下范围内寻找安全位置。
- 移动前异常不改变玩家位置。
- 移动后等待下一更新帧重新验证；失败则回滚原位置并清除危险速度状态。
- 只有二次验证成功才提交，并广播服务器权威 `PositionSet`。
- 集成主机不经过远程 ID 61 的 4 秒截止时间，可使用完整 10 秒区块加载预算。

### 5.3 可诊断错误

- 主机和远程传送都不再使用完全静默的通用异常处理。
- 日志记录请求 ID、路由、失败阶段、异常类型、消息和堆栈；不记录不必要的精确目标坐标。
- 阶段至少区分：区块加载、候选搜索、移动快照、位置写入、下一帧复检、回滚、位置同步。
- 游戏内仍显示简短中文结果，不向普通玩家暴露堆栈；未知异常提示“传送失败，详细原因已写入日志”。
- 超时、无安全落点、被禁用、忙碌、回滚和内部错误保持不同结果，不再全部表现为“没有反应”。

## 6. 测试设计

### 6.1 UI

- 192 逻辑像素在当前 GUI 尺寸下保持右上角可见且不覆盖右侧控制列。
- 设置 schema 迁移只把当前未公开测试配置中的 384 调整为 192，其他合法尺寸不变。
- 玩家传送图标只在联机且存在其他玩家时出现。
- 背包、角色面板、制作台、睡眠、Dialog 和大地图分别验证隐藏与关闭后恢复。
- 小地图点击打开大地图，滚轮缩放和文本输入屏蔽不互相误触。

### 6.2 整区块探索

- 从世界 X/Z 15 跨到 16 时只登记一个新区块；同一区块内移动不重复登记。
- `-1 / -16 / -17` 映射到正确区块。
- 未就绪区块持续保留待办，不写入任何部分像素。
- 从未就绪变为地表可读后，下一次更新恰好采样并提交 256 列。
- 内存地图在磁盘 flush 前即可读取完整 256 像素。
- 重新进入会刷新完整区块，并修复旧透明/残缺数据。
- 不登记、读取或显示尚未进入的邻接区块。

### 6.3 传送

- 使用当前真实 `Survivalcraft.dll` 强制初始化移动状态字段解析，确认公开的 `m_wasStanding` 与非公开的 `m_falling` 均能绑定。
- 字段缺失、静态或非 `bool` 时给出描述性错误。
- 地表和坐标点成功路径各验证只提交一次位置同步。
- 快照前失败不移动；移动后失败回滚；失败路径不广播成功位置。
- 主机、远程 ID 61 与玩家邀请 ID 41 均验证明确结果和诊断阶段。

### 6.4 完整验证

- 运行全部 Release 测试和 `-warnaserror` 构建。
- 生成并执行 `PACKAGE_OK` 封包校验，确认原始 `34GPSFix.netmod` 哈希不变。
- 把新包复制到保留 World2 与旧地图缓存的隔离游戏，完全关闭旧进程后重新启动。

## 7. 人工验收步骤

1. 进入 World2，确认小地图视觉体量接近 256×256 屏幕像素，顶部不再出现文字传送按钮。
2. 打开和关闭背包、角色面板及大地图，确认 HUD 地图自动隐藏和恢复。
3. 从一个 16×16 区块跨入下一区块，确认目标区块一次完整出现，相邻未进入区块仍为空白。
4. 回到旧的残缺/透明区块，确认整块自动刷新，无需删除缓存。
5. 右键已探索地图位置执行地表传送，确认找到安全 Y、玩家移动且日志结果为成功。
6. 新建坐标点，离开后传送回来，确认保存 Y 附近安全落点规则。
7. 联机存在第二名玩家时确认传送图标出现；完成邀请、同意和安全传送。单人时该图标保持隐藏。

## 8. 非目标与约束

- 不删除大地图、缩放、航点、日夜明暗或探索持久化功能。
- 不恢复 Mod 数量扫描或上报。
- 不增加额外大地图文字按钮；入口为 `M` 和点击小地图。
- 不提前揭示仅进入可视范围、但玩家尚未踏入的区块。
- 本设计只参考简洁体量和游戏原生 HUD 关系，不采用固定物理像素实现，也不在公开清单中增加来源或署名说明。
- 第二轮实机验收通过前，不创建公开 release/tag，不宣称修复完成。
