# Survivalcraft Travel Map 用户指南

## 1. 使用前检查

支持目标为 Survivalcraft 联机版 `2.4.40.6`、NetMod API `1.44`、Windows PC。安装前关闭游戏，并确保 `NetMods` 中没有 `34GPSFix.netmod`。两个 Mod 同时存在时 SCTM 会显示冲突提示，不注册网络包，也不初始化玩家地图组件。

不要覆盖原 Mod 文件。将 `SurvivalcraftTravelMap.netmod` 作为独立文件放入 `NetMods`。

## 2. 地图界面

Survivalcraft 联机版的单人世界不是 `WorkType.Local`，而是 `WorkType.Server` 集成主机。集成主机的主玩家仍会创建地图纹理和界面，右上角显示小地图；没有本地主玩家界面的专用服务器或远程服务端玩家实体才不会创建地图 UI，只保留服务器权威传送能力。远程联机客户端的主玩家同样显示地图 UI。

右上角小地图默认逻辑尺寸为 `192`。小地图和邀请图标使用玩家 `GuiWidget.ActualSize` 的逻辑坐标定位，而不是相机物理视口像素。这样在窗口模式和非 100% UI 缩放下，右上角边距仍按游戏界面坐标计算。当前实机验收目标约为 `256×256` 物理像素；该物理尺寸、上/右自适应边距以及不出现文字传送按钮仍需在 UI 缩放 `0.75 / 1.0 / 1.25` 下分别确认。

单击小地图或按 `M` 都会打开大地图，`M` 也可关闭大地图。以下情况下打开操作被主动屏蔽：聊天框/输入法正在输入、任意文本框拥有焦点、或当前有其他模态对话框。这样不会把聊天中的字母 M 当作快捷键。

背包、角色、合成、睡眠、大地图或其他模态界面打开时，小地图和邀请图标会自动隐藏并禁用输入；界面关闭后按原有设置自动恢复，不会改写 `settings.json`。多人邀请使用小型双人传送图标，仅当邀请功能可用且当前还有其他玩家时出现；单人游戏不会显示该图标。

大地图操作：

1. 在地图上滚动滚轮进行缩放，范围为每像素 `0.25..32` 个方块。
2. 按住左键拖动进行平移。
3. 右键已探索普通位置，选择“传送到这里”或“保存当前位置”。后者保存玩家此刻的真实 XYZ，与右键所指的远处地表坐标无关。
4. 右键坐标点，选择“传送到此坐标点”“重命名”或“删除”。
5. 右上角“设置”可调整小地图显示、XYZ、日夜明暗、邀请接收、小地图尺寸和缩放。

小地图尺寸只接受 `160 / 192 / 256 / 320 / 384`，默认值为 `192`。光标悬停小地图且没有聊天/模态输入冲突时，滚轮每格按 `sqrt(2)` 缩放并持久化，范围为每像素 `0.5..8` 个方块；大地图打开时不会同时误缩放小地图。缩放时地图保持世界方向固定，玩家箭头旋转。灰色区域代表未探索，不会暗中读取或显示地形。

## 3. 探索与持久化

探索以玩家进入的 16×16 地形区块为单位。每帧都会观察玩家脚下的世界 X/Z；一旦进入另一个区块就立即调度该区块，只揭示玩家本人进入的区块，不会按可视距离或半径顺带揭示相邻区块。探索更新在地形更新之后运行；当区块达到 surface-readable 阈值 `TerrainChunkState.InvalidPropagatedLight` 或更高状态，并且 256 个地表采样全部得到非透明颜色时，整个 16×16 区块才一次性原子写入，所以地图上会同时出现 256 个探索格，不会逐列显露。

尚未加载到可读地表、缓存压力拒绝新瓦片或采样抛出异常时，该区块都保持 pending 并在后续帧持续重试；每帧最多尝试 4 个区块，且每个区块单独隔离异常，因此一个永久失败的当前区块不会阻塞旧 pending 区块的轮转。缓存压力提示只显示一次；相同区块与相同错误签名的异常日志会节流，后续区块仍继续尝试。地图按 64×64 方块瓦片保存，脏瓦片定期原子写入。损坏的瓦片或坐标点文件会被隔离为 `.corrupt`，不会覆盖其他正常数据。

格式版本 1 把 `A=0` 保留为空气/不可用采样哨兵，实际已探索地形使用非透明颜色。旧版本曾写出的“探索位为真、RGBA 却全透明”像素在读取时会自动视为未探索；玩家离开后再次进入同一区块会重新调度并刷新完整 256 像素，因此保留在 World2 中的 partial legacy cache 也会补全，无需删除整个地图缓存。

数据目录：

```text
app:/SurvivalcraftTravelMap/
  settings.json
  server-settings.json
  maps/
    <world-key>/
      <player-guid>/
        waypoints.json
        tiles/
          <tile-x>_<tile-z>.sctm
```

`world-key` 是 24 位哈希：单人使用规范化世界目录；联机使用服务器主机名/IP、端口和服务器世界标识。`player-guid` 使用 32 位无连字符 GUID。若拿不到可靠服务器端点、世界标识或玩家 GUID，或者初始存储目录、地图瓦片/坐标点持久化无法建立，该玩家组件保持未激活，不创建地图 UI，并显示提示；游戏本身继续运行，也不会向可能错误的玩家目录写入数据。

`settings.json` 使用 `schemaVersion: 1`。首次运行会迁移同目录旧 `travel-map-settings.json`，也会继续迁移旧 `GPSSetting.xml` 的两个兼容开关；有效源文件均保留，损坏的旧 JSON 会隔离并按损坏结果提示，而不会误报迁移成功。更高 schema 只读加载并提示一次，界面修改只影响当前内存，不会写回或覆盖未来版本文件。一般设置读取失败时使用只读安全默认值继续；损坏、`null` 或无效文档会隔离为 `.corrupt` 后创建安全默认值。

瓦片缓存最多保留 128 项。磁盘持续写失败且所有缓存项均为脏数据时，SCTM 不丢弃已有脏瓦片，而是暂停接收新的地图瓦片并提示一次；已有瓦片仍可更新。后续刷新成功后会自动恢复新区块探索，不会在失败期间无限增长内存或启动忙循环。

旧 `GPSSetting.xml` 只迁移“小地图显示”和“接受邀请”两个开关；迁移不会删除旧文件。

## 4. 集成主机安全传送

联机版单人世界以集成主机运行。右键地图位置时，SCTM 先要求该像素已探索，再在同一进程内调用服务器权威安全传送，请求目标周围半径 8 格区块，最长等待 10 秒。集成主机不经过远程 ID 61 的 4 秒协议截止时间。候选点必须满足：

- 脚下为安全实体方块；
- 脚和头所在两格均为空气；
- 不与其他实体碰撞；
- 不使用水、其他流体、树叶、下落方块或伤害方块作为落脚点；
- 坐标位于世界边界内。

通过后玩家被放到 `X+0.5 / Z+0.5` 的方块中心，速度、角速度和坠落伤害状态被清零。下一游戏更新再次验证；若地形或实体状态发生变化，则恢复原位置并保持静止。找不到位置、超时或越界时不会先移动玩家。

坐标点保存创建时的完整浮点 XYZ。搜索顺序先从保存 Y 开始，再按 `+1,-1,...,+8,-8` 检查同一水平列，之后扩大水平距离到半径 8。它不会查询远端地表作为回退。

## 5. 联机与服务器权威

联机地图界面在客户端运行，但坐标传送只能由服务器执行。协议 ID 61 的客户端请求包含请求 ID 和目标，不包含客户端计算的最终安全 Y。服务器使用对应玩家实体重新运行与单人相同的安全搜索和事务回退。

要启用联机地图传送：

1. 客户端和服务器都安装相同版本 SCTM。
2. 服务器首次启动后检查 `app:/SurvivalcraftTravelMap/server-settings.json`。
3. 保持或修改：

```json
{
  "schemaVersion": 1,
  "surfaceTeleportEnabled": true,
  "waypointTeleportEnabled": true
}
```

4. 修改配置后重启服务器。

远程客户端会先发送能力查询。客户端响应窗口为 5 秒，远程服务器对每个 ID 61 执行使用严格更短的 4 秒截止时间。截止或断线的 linked cancellation 会进入完整安全传送回退；因此服务器不会在客户端已经超时后再提交迟到移动。服务器未安装、协议不支持或 5 秒内无响应时，本次连接只显示一次不支持/超时提示。客户端在收到服务器成功结果前不会改变位置。传送通过移动后下一帧复核并正式提交时，服务器广播 `ComponentPlayerPackage.PlayerAction.PositionSet`，使已连接客户端同步权威位置。集成主机不受 ID 61 的远程截止时间约束，仍使用 10 秒区块加载上限。

玩家邀请传送继续使用 ID 41 的旧字段顺序。关闭“接受玩家传送邀请”只会自动拒绝新的邀请弹窗，普通成功/失败消息仍显示。管理员立即传送、30 秒邀请超时和每页 4 人保持兼容。

## 6. 备份、恢复与卸载

备份：

1. 完全退出游戏或服务器，等待进程结束。
2. 复制 `app:/SurvivalcraftTravelMap` 到安全位置。
3. 如需只恢复一个玩家，复制对应 `<world-key>/<player-guid>` 目录。

卸载：

1. 退出游戏。
2. 删除 `NetMods/SurvivalcraftTravelMap.netmod`。
3. 保留数据目录便于以后重装，或确认备份后手动删除。

Mod 不会在卸载时自动删除地图、坐标点、旧 `GPSSetting.xml` 或任何世界文件。

## 7. 故障排查

- 提示与 34GPSFix 冲突：退出游戏，从 `NetMods` 移除旧包，再重启。
- `M` 无反应：先关闭聊天输入、文本框和其他对话框。
- 右上角没有小地图：先在设置中确认“小地图显示”已开启；若 `M` 能打开大地图但右上角仍不可见，请记录窗口大小和 UI 缩放，当前修复仍需实机复测确认。
- 地图为灰色：实际进入该 16×16 区块并等待其达到 `TerrainChunkState.InvalidPropagatedLight` surface-readable 阈值；尚未就绪的区块保持 pending 并重试，不会提前揭示相邻区块。旧透明或 partial legacy cache 会在离开后重入该区块时刷新，不需要删除缓存。
- 传送提示无安全位置：目标可能是水面、洞壁、危险方块或实体拥挤位置；换一个已探索点。
- 坐标点无法传送：保存 Y 附近 ±8 格内没有安全空间；坐标点模式不会改传地表。
- 联机提示服务器不支持：确认服务器安装 SCTM、ID 61 未冲突，且服务器设置开启。
- 地图不保存：检查游戏目录写权限；若出现身份不足提示，重新连接让服务器世界标识和玩家 GUID 完整同步。
- 文件损坏提示：先备份 `.corrupt` 文件；SCTM 会隔离损坏项并继续加载其他数据。

## 8. 隐私与限制

SCTM 不读取 Mod 列表来统计数量，不建立扫描线程，不发送 Mod 数量、校验码或安装清单。启动时唯一的 Mod 身份检查是是否存在不兼容的 `34GPSFix`；网络只注册 ID 41 和 61。

当前已知限制：全细节地图在 64 格瓦片接缝处的青色探索边界可能不连续，但瓦片像素、持久化和传送探索判定不受影响。小地图定位、entered-chunk 原子探索和集成主机传送修复已有自动测试覆盖，但尚未启动当前修复包进行游戏内复测。
